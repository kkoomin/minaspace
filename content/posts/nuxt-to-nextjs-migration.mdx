솔드아웃의 판매/결제 플로우는 Nuxt 2 기반이었습니다. Vue 2 EOL과 레거시 의존성 누적으로 인해 Next.js 마이그레이션을 결정했고, 약 3개월에 걸쳐 완료했습니다.

## 마이그레이션 결정 배경

Nuxt 2는 Vue 2에 의존하는데, Vue 2는 2023년 말 EOL을 맞았습니다. 주요 문제는 세 가지였습니다.

- 보안 패치가 중단된 의존성 다수 포함
- 팀 내 React/Next.js 경험자가 많아 유지보수 부담
- App Router 기반 레이아웃 시스템의 장점 활용 불가

## 페이지 단위 점진적 전환

전체를 한 번에 바꾸지 않고, 트래픽이 낮은 페이지부터 점진적으로 전환했습니다.

### 라우팅 구조 매핑

```
# Nuxt (pages/)
pages/
├── sell/index.vue       → app/sell/page.tsx
├── sell/[id].vue        → app/sell/[id]/page.tsx
└── payment/index.vue    → app/payment/page.tsx
```

### Vuex → Zustand

결제 플로우 상태를 Vuex 모듈에서 Zustand 스토어로 옮겼습니다.

```ts
// Before: Vuex module
export const paymentModule = {
  namespaced: true,
  state: () => ({ step: 1, selectedMethod: null }),
  mutations: {
    SET_STEP(state, step) {
      state.step = step;
    },
  },
};

// After: Zustand
const usePaymentStore = create<PaymentState>((set) => ({
  step: 1,
  selectedMethod: null,
  setStep: (step) => set({ step }),
}));
```

## 트러블슈팅

### SSR hydration 불일치

결제 수단 목록을 서버에서 fetch하면서 hydration mismatch가 발생했습니다. 원인은 서버와 클라이언트의 locale 처리 차이였고, `Intl.NumberFormat`의 locale를 명시적으로 `'ko-KR'`로 고정해 해결했습니다.

```ts
// 수정 전 (암묵적 locale)
const formatted = new Intl.NumberFormat().format(price);

// 수정 후 (명시적 locale)
const formatted = new Intl.NumberFormat("ko-KR").format(price);
```

## 결과

- 초기 로딩 LCP: 3.2s → 1.8s (44% 개선)
- JS 번들 크기: 220kb → 145kb (34% 감소)
- 빌드 타임: 4분 → 1분 30초
